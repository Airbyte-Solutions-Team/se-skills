## Summary

Wires the existing Salesforce and Gong MCP invocation patterns into `pov-gsheet` through a thin, deterministic bridge, removes the need for SEs to manually prepare evidence JSON, and adds existing-sheet detection and consent/permissions disclosure to the skill. The Google Sheets/Drive end-to-end run and live MCP smoke tests cannot be completed in this environment because no MCP servers are configured and Chrome is not signed into Google.

## What changed

- Added `webapp/pov_gsheet_bridge.py` — a standalone `uv` script that converts raw MCP/tool output into the `ExternalEvidence` JSON already consumed by `webapp/pov_gsheet_context.py`.
- Updated `skills/pov-gsheet/SKILL.md` so the skill itself calls the configured MCP tools, writes raw responses to a job-scoped temp directory, runs the bridge, and passes the resulting evidence files to the deterministic loader. SEs no longer prepare evidence JSON by hand.
- Reused the established Salesforce and Gong patterns from `skills/_se-playbook.md`:
  - Salesforce: `mcp__salesforce__run_soql_query` (read-only SOQL) scoped by `Account.Name LIKE`, picking the most relevant open non-renewal opportunity.
  - Gong: `mcp__gong__search_calls` with account/domain + bounded date filters, then `gong://calls/{callId}/transcript`.
- Added source/destination classification in the bridge so `technical_system` facts end up in the right `PovContext` bucket.
- Enhanced `webapp/scripts/pov-gsheet-runner.mjs` with Google sign-in detection, an existing-sheet check, Drive-placement verification, and installed Playwright in `webapp/scripts`.
- Added `eval/tests/test_pov_gsheet_bridge.py` with deterministic tests for Salesforce/Gong normalization, source/destination classification, malformed input, unavailable-source reporting, CLI invocation, and `PovContext` hand-off.
- Updated `IMPLEMENTATION-PLAN.md` (POV-001 remains `Partial`, POV-002 is `In Progress`), `README.md`, and `webapp/SESSION-LOG.md`.

## How MCP results enter `ExternalEvidence` and `PovContext`

```text
Skill (claude -p)
├── calls mcp__salesforce__run_soql_query
│   └── writes $JOB_DIR/pov-gsheet-salesforce-raw.json
├── calls mcp__gong__search_calls + gong://calls/{id}/transcript
│   └── writes $JOB_DIR/pov-gsheet-gong-search.json
│       └── $JOB_DIR/pov-gsheet-gong-transcripts.json
└── runs webapp/pov_gsheet_bridge.py --source {salesforce|gong} ...
    └── writes $JOB_DIR/pov-gsheet-{source}-evidence.json

webapp/pov_gsheet_context.py --{source}-evidence ...
├── validates each ExternalEvidence
├── merges prospect, contacts, business objectives, technical systems, requirements, success criteria, milestones
├── deduplicates and surfaces conflicts
└── emits PovContext JSON + source_coverage list
```

The bridge never invents data and never calls an MCP itself; it only normalizes the raw tool output the skill already produced.

## What no longer requires manual evidence preparation

- The SE does not write `ExternalEvidence` JSON files.
- The SE does not manually query Salesforce or Gong.
- The skill produces raw MCP responses and the bridge turns them into the loader's expected format.

## Source handling

- **Salesforce** and **Gong** are classified as `searched` only when the MCP tool is actually invoked; if the MCP is not configured the bridge writes `status: unavailable` with an explicit note.
- **Granola / Gmail / Slack** are checked; if no usable MCP exists the bridge writes `status: unavailable` and the skill records them in `source_coverage` without claiming they were searched.
- Local transcripts and prior outputs remain fully operational.

## Google Sheets/Drive verification status

- The Playwright helper has sign-in detection, existing-sheet detection, title verification, and Drive-folder placement verification.
- A real run cannot be performed because Chrome is not signed into a Google account (`https://drive.google.com/drive/home` redirects to Google sign-in).
- The deterministic dry-run plan (TSV payloads for all seven tabs, dropdown validation, copy URL construction) is verified by tests.

## What appears in the receipt

- Final Google Sheet URL and Drive folder URL (when available).
- `source_coverage` entries with `source`, `available`, `material`, `status`, `freshness`, and `note`.
- `tabs_populated` list and any `warnings`.
- `status` (`created`, `created with gaps`, `blocked`, `run-error`, or `existing-sheet-detected`).

## What the user should manually test

1. Configure an MCP server for Salesforce and/or Gong in `~/.claude.json`.
2. Ensure `salesforce.enabled: true` (for the `sf` CLI fallback) and the `pov_gsheet` block in `~/.se-config.yaml` points to a real template and Drive folder.
3. Sign Chrome into the Google account that owns the template and Drive folder.
4. Run `pov-gsheet` against a safe test prospect and verify:
   - all seven tabs are populated,
   - contacts, business objectives, success criteria, and in-scope apps match the context,
   - dropdown values are valid,
   - the sheet lands in the correct prospect subfolder,
   - the title is `Airbyte || [Prospect Name] - POV Success Criteria`,
   - the receipt accurately lists sources as `searched`, `unavailable`, or `skipped`.

## What could break

- The bridge's Salesforce/Gong parsing relies on common response shapes; unusual MCP response formats will be treated as `unavailable` rather than silently mis-parsed.
- Gong connector classification is heuristic; complex sentences may misclassify a system as source/destination.
- The Playwright helper selectors for Drive and Sheets UI may drift; it degrades to manual instructions and a dry-run plan.
- `uv` must be on PATH for the skill to run the bridge and loader reliably.

## What remains externally dependent

- Live Salesforce MCP server and Gong MCP server (none configured in this environment).
- A signed-in Chrome profile with access to the POV template and Drive folder.
- The template must contain the seven named tabs and the dropdown lists used by `pov-gsheet-runner.mjs`.
